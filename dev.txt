DEPLOYMENT RUNBOOK — go-dbms Backend + React Frontend on Oracle Cloud
======================================================================

1. SSH INTO THE SERVER
-----------------------
ssh -i ~/.ssh/oracle-dms-key.pem ubuntu@140.238.79.81


2. BACKEND — DOCKER CONTAINERS
--------------------------------
Start everything:
    cd ~/go-dbms
    sudo docker compose up -d

Check status:
    sudo docker ps -a

Should see 4 containers healthy/running:
    go-dbms-backend-1   (4000)
    go-dbms-postgres-1  (5430 -> 5432)
    go-dbms-clamd-1     (3310)
    go-dbms-minio-1     (9000-9001)

If a container shows "Exited":
    sudo docker start <container-name>

View backend logs (debugging):
    sudo docker logs go-dbms-backend-1 --tail 100 -f


3. FRONTEND — REACT (VITE + PNPM)
-----------------------------------
One-time setup (Node, pnpm):
    node --version
    sudo corepack enable
    corepack prepare pnpm@8.10.0 --activate
    pnpm --version

Install dependencies:
    cd ~/react-dbms
    pnpm install

Set the backend API URL (must be the server's public IP, not localhost):
    echo 'VITE_API_URL=http://140.238.79.81:4000/dbms/v1' > .env.production

Build (required every time .env.production or source code changes —
Vite bakes env vars in at BUILD time, not run time):
    pnpm build

Serve the build:
    pkill -f "vite preview"    # stop any previous instance first
    nohup npx vite preview --host 0.0.0.0 --port 3000 > preview.log 2>&1 &

Note: --host 0.0.0.0 is required. Without it, Vite only listens on
localhost and stays unreachable from outside the server.


4. OPEN PORTS (needed once per port, persists after)
------------------------------------------------------
Every exposed port needs BOTH of the following, or it stays unreachable.

a) OCI Console (browser):
   Networking -> Virtual Cloud Networks -> your VCN -> Subnet ->
   Security List (or NSG if attached to the instance) -> Add Ingress Rule
     Source CIDR: 0.0.0.0/0 (tighten later)
     Protocol: TCP
     Ports needed: 4000 (backend API), 3000 (frontend)
   Do NOT expose 5430 (Postgres) or 3310 (clamd) publicly.

b) iptables on the instance (terminal):
    sudo iptables -L INPUT -n --line-numbers     # inspect current rules first
    sudo iptables -I INPUT -p tcp --dport 4000 -j ACCEPT
    sudo iptables -I INPUT -p tcp --dport 3000 -j ACCEPT
    sudo netfilter-persistent save               # persist across reboot


5. VERIFY
----------
    curl -I http://localhost:4000/dbms/v1/auth/login    # backend responds locally?
    curl -I http://localhost:3000                        # frontend responds locally?

Then in a browser:
    Frontend: http://140.238.79.81:3000
    Backend (JSON API only, no browsable UI): http://140.238.79.81:4000/dbms/v1/...


COMMON GOTCHAS HIT SO FAR
---------------------------
- "pnpm run preview -- --host ... --port ..." inserts a stray "--" that
  breaks Vite's flag parsing. Call "npx vite preview --host 0.0.0.0
  --port 3000" directly instead.

- Frontend defaults to http://localhost:4000 in src/services/api.ts if
  VITE_API_URL isn't set before building. "localhost" in that context
  means the BROWSER's machine, not the server, causing
  ERR_CONNECTION_REFUSED.

- "corepack enable" needs sudo since Node was installed system-wide.

- Foreground processes (pnpm preview without nohup) die when the SSH
  session ends. Always background long-running servers with nohup + &.

- "docker compose down -v" removes volumes, not just containers — this
  wipes the Postgres data (go-dbms_dms_pgdata) and MinIO data
  (go-dbms_dms_minio) permanently. Only ever run plain "docker compose
  down" (no -v) unless you explicitly intend to destroy all data.
  RUN_MIGRATION=true will happily recreate the empty schema on next
  boot, which masks the data loss — the app looks "fine", just empty.

- "pkill -f 'vite preview'" run over SSH can self-match: the pattern
  you're searching for is literally present in the remote shell's own
  command line (...pkill -f 'vite preview'...), so pkill kills its own
  invocation and drops the SSH session before finishing. Kill by port
  instead: "sudo lsof -ti:3000 | xargs -r kill -9" (or -ti:4173 if it's
  stuck on Vite's default port from the stray "--" gotcha above).

2026-07-07 — REGISTRATION BROKEN: CORS ONLY ALLOWED localhost:3000
----------------------------------------------------------------------
Symptom: user could not register from the real deployed site. curl
against the API worked fine (201, wrote to Postgres) — the confusing
part is curl doesn't enforce CORS, so it looked like the backend was
healthy while the actual browser UI was fully blocked.

Root cause: cmd/server/main.go's CORS allowlist hardcoded only
"http://localhost:3000". The deployed frontend's real origin is
"http://140.238.79.81:3000", which isn't in that map, so every
browser fetch (register, login, everything) failed the CORS preflight
with net::ERR_FAILED before it even reached the backend.

Fix: added "http://140.238.79.81:3000" to the allowedOrigins map.
Verify a fix like this with a real preflight, not curl:
    curl -X OPTIONS http://140.238.79.81:4000/dbms/v1/auth/register \
      -H "Origin: http://140.238.79.81:3000" \
      -H "Access-Control-Request-Method: POST" -D - -o /dev/null
  -> should show "Access-Control-Allow-Origin: http://140.238.79.81:3000"

Deployed via the normal push/pull cycle, then rebuilt the backend
image with no cache to be sure the fix was actually in the running
container:
    cd ~/go-dbms && git pull origin main
    sudo docker compose build --no-cache backend
    sudo docker compose up -d --force-recreate backend

Frontend (video splash-screen + cardless auth pages) was deployed the
same way: git pull, pnpm install, pnpm build, kill whatever's on
:3000 by port (not pkill -f, see gotcha above), nohup npx vite preview
--host 0.0.0.0 --port 3000.

If CORS breaks again after a future domain/IP change, this map is the
first place to check — there's no wildcard, every origin must be
listed explicitly.
